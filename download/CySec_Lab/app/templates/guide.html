Authentication Easy Challenge 1: Authentication Bypass

The login form is vulnerable to classic SQL injection. By crafting a malicious input in the username or password field, you can bypass authentication entirely and gain unauthorized access to the application.

🎯 Objective

Log in as the admin user without knowing the real password. The backend constructs the query as:

SELECT * FROM users WHERE username = '{user_input}' AND password = '{pass_input}'

Find an input that makes this query always evaluate to TRUE.

💡 Show Hints
  • Think about how SQL handles string termination with single quotes.
  • Try injecting ' OR '1'='1 or ' OR 1=1 --
  • Pay attention to comment syntax (--, #, /*) to truncate the rest of the query.
  • On Medium level, basic keyword filters are applied — try case variations or URL encoding.
  • On High level, prepared statements partially replace raw queries — look for remaining injection points.

Injection Easy Challenge 2: Error-Based SQL Injection

The product search feature directly concatenates user input into an SQL query. Deliberately crafted input will trigger database errors that leak sensitive information such as table names, column names, and database version.

🎯 Objective

Extract the database version and the name of the database management system by triggering informative error messages.

SELECT * FROM products WHERE name LIKE '%{user_input}%'
💡 Show Hints
  • Use a single quote to break the string context and observe the error message.
  • Functions like EXTRACTVALUE(), UPDATEXML(), or JSON_KEYS() can trigger verbose errors.
  • Example: ' AND EXTRACTVALUE(1, CONCAT(0x7e, VERSION())) --
  • The number of columns in the result set matters — use ORDER BY to enumerate them.

Injection Medium Challenge 3: UNION-Based SQL Injection

Using the UNION SQL operator, you can append an additional query to the original and retrieve data from other tables within the same database. This challenge requires you to determine the column count and map your injection to displayable columns.

🎯 Objective

Extract the list of usernames and passwords from the users table.

SELECT product_id, name, price FROM products WHERE name LIKE '%{user_input}%'
💡 Show Hints
  • First, find the number of columns using ORDER BY N-- — increment N until you get an error.
  • Then use UNION SELECT 1,2,3-- (adjust column count) to find displayable columns.
  • Replace the displayable columns with username, password FROM users.
  • Use information_schema.tables to enumerate tables if needed.
  • On higher security levels, the word UNION may be filtered — try mixed case like UnIoN.

Blind Medium Challenge 4: Boolean-Based Blind SQL Injection

The user lookup page returns different responses depending on whether a user exists or not, but never reveals data directly. You must infer information character by character using TRUE/FALSE conditions in your injected SQL.

🎯 Objective

Extract the admin user's password hash one character at a time using conditional queries.

SELECT * FROM users WHERE user_id = {user_input}
💡 Show Hints
  • Test with 1 AND 1=1 (should return results) vs 1 AND 1=2 (should return nothing).
  • Use SUBSTRING(password, N, 1) = 'X' to test each character of the hash.
  • Use ASCII() and binary search to speed up character extraction.
  • Automate with a script — humans shouldn't manually enumerate 32+ characters.
  • Try sqlmap with --technique=B for automated boolean-based extraction.

Blind Hard Challenge 5: Time-Based Blind SQL Injection

The network diagnostic tool uses user-supplied IDs in a query but returns no visible output — only the response time differs. Inject conditional SLEEP() or BENCHMARK() calls to infer data by measuring response delays.

🎯 Objective

Determine the database name and extract at least one credential using only timing differences as your oracle.

SELECT response_text FROM diagnostics WHERE target_id = {user_input}
💡 Show Hints
  • Test with 1; SELECT SLEEP(5)-- — if the response takes ~5 seconds, injection is possible.
  • Use conditional timing: 1 AND IF(SUBSTRING(DATABASE(),1,1)='c', SLEEP(5), 0)--
  • Each character test takes one request — this is slow. Use binary search to reduce attempts.
  • Tools like sqlmap --technique=T can automate this process.
  • Be aware of network jitter — repeat tests to confirm timing is consistent.

Second-Order Hard Challenge 6: Second-Order SQL Injection

Unlike traditional SQLi, the malicious payload is stored in the database during registration (a seemingly safe operation) and later executed when a different page queries that stored data. The vulnerability exists not in the registration form but in the profile search feature.

🎯 Objective

Register a new account with a crafted username, then trigger the stored payload via the profile search to extract admin credentials.

💡 Show Hints
  • The registration page sanitizes input, but the profile search page does not properly parameterize its query.
  • During registration, store a payload in a field like the department or username.
  • Example: register with username normal_user' -- or a payload in the department field.
  • Then navigate to the profile search page and look up that user — the stored data becomes part of a new SQL query.
  • This is one of the hardest injection types to detect and prevent.

Privilege Escalation Hard Challenge 7: Privilege Escalation via SQL Injection

The admin panel allows role updates for users, but the role assignment endpoint is vulnerable. By exploiting SQL injection in the user_id or role parameter, you can escalate your own privileges from a regular user to an administrator.

🎯 Objective

Elevate your user account's role to admin using SQL injection in the role update feature.

UPDATE users SET role = '{user_input}' WHERE user_id = {id}
💡 Show Hints
  • The vulnerability may be in either the user_id or new_role parameter.
  • If injecting into user_id, try stacking queries or modifying the WHERE clause.
  • Consider: 1 OR 1=1 to update ALL users, or target a specific condition.
  • Stacked queries (;) may work depending on the database driver and Python connector.
  • On higher levels, the app may validate role values — look for ways to bypass the validation logic.
