💡 Challenge Hints — Second-Order SQL Injection

Key Concept: Unlike traditional SQL injection, second-order injection exploits data that is stored in the database and later used unsafely in a different query.

How It Works

  1. Step 1 — Store the payload: Register an account and embed SQL injection payload into one of the fields (username, email, or department). The registration page may appear to handle input safely.
  2. Step 2 — Trigger the payload: Navigate to the Profile Search page and look up your user. The stored data is now used in a vulnerable query, causing your payload to execute.

Payload Ideas

  • Register with a username containing SQL: test' OR '1'='1
  • Store a payload in the department field: Engineering' UNION SELECT 1,username,password,4,5 FROM users--
  • Close quotes strategically to alter the profile search query structure.

Why This Is Dangerous

Second-order injection is particularly insidious because:

  • The input point (registration) appears safe and passes security reviews.
  • The execution point (profile search) uses trusted database data, so developers may not apply input sanitization.
  • It often evades Web Application Firewalls (WAFs) that only inspect the initial request.
